AS9100D ERP FRONTEND PAGE AND COMPONENT MAP
Date: 2026-03-08

====================================================================
1. PURPOSE
====================================================================

This document is the next planning step after the frontend implementation
guide. It translates the backend structure into a concrete frontend page map
and reusable component structure.

Use this document to decide:

1. What pages need to exist.
2. What each page should contain.
3. Which components should be shared.
4. What should be built first in React or Next.js.
5. How module pages should connect to each other.


====================================================================
2. TARGET FRONTEND MODEL
====================================================================

Recommended stack assumptions:

1. React or Next.js frontend
2. TypeScript
3. API client wrapper around fetch or axios
4. Role-aware routing
5. Shared app shell
6. Shared table, form, workflow, and document components

This plan works for both:

1. React Router app
2. Next.js app router or pages router

====================================================================
6. SHARED CORE COMPONENTS
====================================================================

Build these shared components once and reuse everywhere.

6.1 StatusBadge
Used for:
1. order status
2. inspection status
3. approval state
4. active or inactive state

6.2 DataTable
Used for list pages.
Needs:
1. sorting
2. filtering
3. pagination
4. row actions
5. loading state
6. empty state

6.3 DetailHeader
Used on detail pages.
Should show:
1. title
2. primary identifier
3. status badge
4. action buttons
5. secondary metadata

6.4 WorkflowStepper
Used for document lifecycle modules such as sales and dispatch.

6.5 ActionPanel
Used on detail pages where workflow actions are important.

6.6 EntitySummaryCard
Shows key metadata in small blocks.

6.7 TimelinePanel
Shows chronological events, revisions, or history.

6.8 AuditInfo
Shows created at, updated at, created by, updated by.

6.9 PdfDownloadButton
Standardized document download action.

6.10 RelatedRecordsPanel
Shows cross-module links.

6.11 ConfirmActionDialog
Used for approve, release, complete, close, and ship actions.

6.12 FormSection
Reusable layout wrapper for grouped form fields.

6.13 LoadingState
6.14 EmptyState
6.15 ErrorState

====================================================================
19. CROSS-MODULE LINKING RULES
====================================================================

Every detail page should link to related downstream or upstream records.

Important links:

1. Customer -> Enquiries, Quotations, Sales Orders
2. Sales Order -> Route Card, Purchase Order, Production Order, Dispatch Order
3. GRN Batch -> Inventory -> Production Log -> Traceability
4. Production Order -> Inspections, Rework, FAI, Dispatch
5. NCR -> Root Cause -> CAPA
6. Machine -> Breakdown -> Work Order -> Downtime
7. Dispatch -> Invoice, Challan, Production Order, CoC

Do not build detail pages as isolated screens.

====================================================================
23. FINAL SUMMARY
====================================================================

This page/component map is the frontend translation of the backend ERP design.

Use it as the next planning layer after the implementation guide.

Most important rule:

Build by workflow and related records, not by isolated CRUD pages.

If you respect this, the frontend will fit the backend correctly and will be
much easier to scale as more modules are added.
