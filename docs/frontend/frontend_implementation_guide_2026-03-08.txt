AS9100D ERP FRONTEND IMPLEMENTATION GUIDE
Date: 2026-03-08

====================================================================
1. PURPOSE
====================================================================

This document explains the backend structure of the ERP project from a
frontend development point of view.

It is meant to help frontend implementation by answering these questions:

1. What is the overall project structure?
2. How do the business modules connect to each other?
3. What screens should the frontend contain?
4. What workflows should the frontend respect?
5. What shared UI patterns should be implemented early?
6. In what order should the frontend be built?


====================================================================
2. PROJECT OVERVIEW
====================================================================

This ERP backend is built with FastAPI and follows a manufacturing workflow
aligned with AS9100D-style business processes.

The main business chain is:

1. Sales creates demand.
2. Engineering defines product and process instructions.
3. Purchase procures required material.
4. Stores receives and verifies material.
5. Production executes manufacturing.
6. Quality verifies conformance and traceability.
7. Dispatch releases finished goods and documents.
8. Maintenance supports machines and can block production.
9. Admin provides system-wide visibility, search, audit, and analytics.

This means the frontend should not be designed as isolated CRUD modules.
It should be designed as connected workflow domains.


====================================================================
3. BACKEND HIGH-LEVEL STRUCTURE
====================================================================

Important backend areas:

1. app/main.py
   - FastAPI application entry point.
   - Handles startup tasks.

2. app/api/v1/api.py
   - Main router registration point.
   - Connects all modules under /api/v1.

3. app/api/deps.py
   - Authentication and authorization helpers.
   - Role checks and module access enforcement.

4. app/api/v1/endpoints/
   - Shared endpoint files for older modules.
   - Includes auth, users, roles, sales, engineering, purchase, stores,
     production, maintenance.

5. app/modules/
   - Business module directory.
   - Newer modules are more self-contained.

Current module folders:

1. admin
2. dispatch
3. engineering
4. maintenance
5. production
6. purchase
7. quality
8. sales
9. stores

====================================================================
5. AUTHENTICATION AND ACCESS CONTROL
====================================================================

Authentication model:

1. Bearer access token for protected requests.
2. Refresh-token-based session continuation.
3. Admin-controlled users and roles.
4. Module access can be explicitly assigned to roles.

Important authorization behavior:

1. Admin can access all protected areas.
2. Non-admin roles may be allowed by role name.
3. If module access entries exist for a role, access is enforced by module key.
4. Locked, inactive, deleted users are blocked.
5. Users linked to inactive roles are blocked.

Frontend requirements:

1. Use one centralized auth provider.
2. Add bearer token automatically to protected API requests.
3. Handle 401 by refreshing or redirecting to login.
4. Handle 403 by hiding or blocking inaccessible pages.
5. Build menu visibility from role and module access.


====================================================================
6. BACKEND MODULE FILE PATTERN
====================================================================

Newer modules follow a clean domain pattern:

1. models.py
   - Database entities.

2. schemas.py
   - Request and response payloads.

3. services.py
   - Business rules.

4. routers.py
   - API endpoints.

5. reports.py or traceability.py
   - PDF/report logic or domain helper logic.

Frontend mirror pattern should be:

1. api/
   - Endpoint call wrappers.

2. types/
   - DTO types and UI types.

3. features/
   - Screen and domain logic grouped by module.

4. components/
   - Reusable UI blocks.

5. guards/
   - Route protection and permission checks.

11. OVERALL BUSINESS DEPENDENCY MAP
====================================================================

This is the most important dependency map for frontend planning:

1. Dispatch depends on Quality and Production.
2. Production depends on Engineering and Stores.
3. Stores depends on Purchase.

5. Maintenance can block Production.
6. Admin reads across all modules.

Frontend implementation should follow the same direction.

====================================================================
14. SHARED FRONTEND DATA HANDLING RULES
====================================================================

Implement one shared API client with:

1. bearer token injection
2. refresh handling
3. common error translation
4. consistent data fetching and mutation patterns

Recommended response handling:

1. 400 -> business rule error shown as user-facing validation message
2. 401 -> refresh or redirect to login
3. 403 -> show access denied and hide action buttons
4. 404 -> show not found state
5. 422 -> field-level validation mapping

====================================================================
17. WHAT THE FRONTEND MUST RESPECT
====================================================================

1. Status-driven workflows matter more than plain CRUD.
2. Many modules depend on prior module completion.
3. Several screens need document behavior and PDF download support.
4. Traceability and related-record linking are essential.
5. Machine maintenance state affects production state.
6. Dispatch readiness depends on multiple upstream validations.


====================================================================
18. FINAL DEVELOPMENT ADVICE
====================================================================

If you want the frontend to fit this backend cleanly:

1. Build by workflow, not just by table.
2. Reuse shared page patterns.
3. Show blockers explicitly.
4. Make related records clickable across modules.
5. Keep permissions centralized.
6. Treat PDFs and reports as first-class features.

Most important summary:

Dispatch depends on Quality and Production.
Production depends on Engineering and Stores.
Stores depends on Purchase.
Purchase and Engineering depend on Sales.
Maintenance can block Production.
Admin reads everything.

If the frontend respects this chain, it will match the backend correctly.
